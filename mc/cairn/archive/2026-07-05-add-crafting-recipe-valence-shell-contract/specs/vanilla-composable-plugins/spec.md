## ADDED Requirements

### Requirement: Crafting recipe Valence shell inventory

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.inventory] Crafting recipe Valence shell-contract work MUST inventory the accepted selected-matrix crafting artifacts, selected-matrix evidence boundary, and relevant Valence plugin/schedule contract sources before defining a runtime shell boundary.

#### Scenario: Shell prerequisites are reviewable

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.inventory.reviewable]
- GIVEN crafting recipe shell-contract work starts
- WHEN reviewers inspect the inventory
- THEN the behavior card, selected-matrix core, validated fixture, receipt handoff, target Java/protocol scope, selected shaped row, selected shapeless row, invalid/no-result row, primary-click collection boundary, and relevant Valence plugin/schedule contract requirements are named
- AND unresolved runtime API, inventory ownership, packet-event, or schedule facts are recorded as implementation-time inspection items instead of assumed behavior.

### Requirement: Crafting recipe Valence shell contract

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract] Crafting recipe runtime work MUST define a Valence Bevy/ECS shell contract before adding any crafting-table shell system, plugin registration, schedule wiring, inventory mutation adapter, target-version data loader, or default plugin membership.

#### Scenario: Shell boundary is explicit

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.boundary]
- GIVEN the selected-matrix core and receipt handoff exist
- WHEN reviewers inspect the shell contract
- THEN it maps future ECS snapshots of crafting-grid state, output-slot state, selected recipe rows, collection requests, client/inventory ownership, and target inventory slots to selected-matrix core inputs
- AND it maps core outputs to shell-owned mutations or diagnostics
- AND it names planned opt-in plugin ownership, resources, components, events, candidate schedule phase, ordering dependencies, disabled-plugin behavior, data-loading boundaries, packet/logging boundaries, and mutation boundaries.

#### Scenario: Shell contract remains bounded

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.bounded]
- GIVEN a shell contract is accepted
- WHEN later crafting work uses it
- THEN the contract permits only selected-matrix runtime planning until separate evidence broadens scope
- AND it explicitly rejects DefaultPlugins membership changes, all-recipe breadth, arbitrary collection modes, shift-click/drag/split handling, data-pack loading, recipe-book UI behavior, recipe discovery or advancement behavior, automated crafter behavior, broad vanilla parity, public-server safety, and production readiness.

### Requirement: Crafting recipe Valence shell contract validation

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.validation] Shell-contract work MUST include focused positive and negative validation for contract completeness, core/shell separation, schedule facts, disabled-plugin behavior, test coverage, evidence requirements, and overclaim rejection.

#### Scenario: Complete shell contract passes

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.validation.positive]
- GIVEN a shell contract records target scope, selected-matrix prerequisites, core input/output mapping, shell-owned resources/components/events, inventory/client ownership, schedule facts, disabled-plugin behavior, positive tests, negative tests, evidence requirements, and non-claims
- WHEN the focused contract validator runs
- THEN it passes with deterministic diagnostics naming the accepted shell boundary.

#### Scenario: Incomplete or overbroad shell contract fails

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.validation.negative]
- GIVEN a shell contract is missing target scope, selected-matrix prerequisites, core/shell separation, shell ownership, inventory/client ownership, schedule facts, disabled-plugin behavior, positive or negative tests, evidence requirements, or required non-claims
- OR it claims DefaultPlugins membership changes, all-recipe breadth, arbitrary collection-mode breadth, data-pack loading, recipe-book behavior, automated crafter behavior, broad vanilla parity, public-server safety, or production readiness
- WHEN the focused contract validator runs
- THEN it fails with a diagnostic naming the missing boundary or rejected overclaim.

### Requirement: Crafting recipe Valence shell contract documentation

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.docs] Crafting selected-matrix documentation MUST identify the shell contract as the required prerequisite before Valence runtime behavior claims.

#### Scenario: Runtime prerequisites are visible

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.docs.reviewable]
- GIVEN reviewers inspect crafting selected-matrix docs after shell-contract work
- WHEN they compare behavior-card, core, fixture, receipt-handoff, and shell-contract docs
- THEN they can identify which artifacts prove local selected-matrix semantics, which artifacts bridge selected receipt evidence, and which shell-contract prerequisites remain before runtime integration.

### Requirement: Crafting recipe Valence shell contract closeout

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.closeout] Shell-contract work MUST record baseline fixture/core/receipt-handoff validation, focused contract validation, Cairn proposal/design/tasks gates, Cairn validation, task-evidence validation, accepted-spec sync verification, evidence-manifest checks, and schedule checks when plugin wiring or schedule registration changes.

#### Scenario: Shell contract closeout is reviewable

r[vanilla_composable_plugins.crafting_recipe_valence_shell_contract.closeout.log]
- GIVEN shell-contract work is ready to archive
- WHEN reviewers inspect task evidence
- THEN successful logs show baseline crafting checks, positive and negative contract validation, Cairn gates, Cairn validation, task-evidence validation, accepted spec requirement IDs, evidence-manifest freshness, and any required Valence schedule evidence
- AND the evidence preserves non-claims for Valence runtime integration until separately implemented, default plugin membership, broad Minecraft compatibility, broad vanilla parity, all recipes, arbitrary collection modes, data packs, recipe-book behavior, public-server safety, and production readiness.
