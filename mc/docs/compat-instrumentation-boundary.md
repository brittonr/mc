# Compatibility instrumentation boundary

This inventory separates harness instrumentation from core Minecraft client/server behavior. It is review guidance for `separate-compat-instrumentation`; it does not claim broader protocol support, production readiness, public-server safety, full CTF/survival correctness, or semantic equivalence.

## Boundary contract

Compatibility instrumentation is opt-in. The harness enables it through one of these boundaries:

- Stevenarella MCP control and frame capture: `clients/stevenarella/src/compat_instrumentation.rs`, with CLI entrypoints `--mcp-stdio`, `--mcp-listen`, `--mcp-token-env`, `--capture-dir`, and capture-recording flags. Default client startup creates no MCP command queue or MCP capture request queue.
- Stevenarella scenario probes: `clients/stevenarella/src/server/mod.rs` and `clients/stevenarella/src/world/mod.rs`, enabled by runner-owned `MC_COMPAT_*` environment toggles and consumed by `compat/runner/src/scenario_catalog.rs`/typed-event fixtures.
- Valence CTF fixture probes: `servers/valence/examples/ctf.rs`, enabled by runner-owned `MC_COMPAT_*` environment toggles and emitted as `MC-COMPAT-MILESTONE` lines.
- Valence survival fixture probes: `servers/valence/examples/survival_compat.rs`, enabled by runner-owned fixture environment toggles and emitted as `MC-COMPAT-MILESTONE` lines.
- Runner orchestration and receipt instrumentation: thin binary entrypoint `compat/runner/src/main.rs`, application shell `compat/runner/src/lib.rs`, `compat/runner/src/scenario_core.rs`, `compat/runner/src/scenario_catalog.rs`, and `compat/runner/src/scenario_manifest_generated.rs`.

Event names and milestone IDs are evidence vocabulary. Moving code must keep names stable unless runner fixtures/checkers migrate in the same change.

## Validation expectations

- Client boundary changes: run focused Stevenarella MCP/instrumentation tests through the mc devshell, then an affected dry-run such as `mc-compat-mcp-controlled-smoke -- --dry-run` when MCP vocabulary is relevant.
- Runner generated surfaces: run scenario manifest/generated harness checks only when the manifest or generated runner tables change.
- Evidence closeout: copy logs under `docs/evidence/`, include explicit `exit_status=0`, and cite a BLAKE3 manifest before checking Cairn tasks.
