# Bevy lifecycle cleanup inventory

## Scope

This change selects cleanup paths already owned by Valence client/inventory lifecycle code. It does not use Hyperion code or concepts, does not change Valence's explicit `Despawned` model, does not add or reorder cleanup `SystemSet`s, and does not claim broad Minecraft compatibility, vanilla parity, public-server safety, production readiness, full CTF correctness, or full survival correctness.

## Explicit cleanup that remains intentional

`OpenInventory` packet notification, missing backing-inventory closure, and `SurvivalOpenContainer` despawn cleanup remain explicit. Their key spaces are client entities and referenced inventory entities, their cleanup triggers are component removal, missing target entity, or `Despawned`, and stale entries are either impossible after component removal or covered by negative tests. They are not migrated to automatic component ownership because each path needs packet emission, cross-entity reference validation, or Valence's explicit despawn window.
